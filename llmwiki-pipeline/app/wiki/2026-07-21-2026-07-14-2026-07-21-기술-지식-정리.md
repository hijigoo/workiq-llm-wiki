## Kubernetes CrashLoopBackOff 트러블슈팅

### 핵심 절차

`CrashLoopBackOff`가 발생하면 아래 순서로 원인을 추적합니다.

1. Pod 이벤트 확인
2. 이전 컨테이너 로그 확인
3. 종료 원인에 따라 리소스 또는 프로브 설정 점검

원문에 포함된 절차는 다음과 같습니다.

```text
kubectl describe pod로 이벤트 확인 -> kubectl logs --previous로 직전 컨테이너 로그.
OOMKilled면 requests/limits 재조정.
readinessProbe 타임아웃이 잦으면 initialDelaySeconds 상향.
```

### 운영 팁

#### `OOMKilled`가 보이는 경우
- 메모리 부족으로 종료된 상황으로 해석
- 컨테이너 리소스의 `requests/limits` 재조정을 우선 검토

#### `readinessProbe` 타임아웃이 잦은 경우
- 애플리케이션 초기 기동 시간이 프로브 설정보다 긴지 확인
- 필요 시 `initialDelaySeconds` 상향 검토

### 주의사항

- 본 자료에는 구체적인 예시 YAML, 명령 전체 형식, 추가 진단 항목은 포함되어 있지 않습니다.
- `livenessProbe`, 애플리케이션 시작 실패, 이미지/시크릿 오류 등 다른 원인에 대한 상세 절차는 이번 수집 범위에서 확인되지 않았습니다.

## PostgreSQL 커넥션 풀 튜닝 Lessons Learned

### 핵심 내용

PostgreSQL 연결 수를 직접 늘리기보다, PgBouncer를 활용한 멀티플렉싱이 더 안정적이었다는 운영 경험이 공유되었습니다.

원문 요약은 다음과 같습니다.

```text
PgBouncer를 transaction 모드로 두고
app 쪽 pool_size는 CPU 코어수*2로 제한.

idle_in_transaction_session_timeout=30s로 좀비 커넥션 방지.

max_connections를 무작정 늘리기보다
PgBouncer로 멀티플렉싱하는 게 안정적이었음.
```

### 권장 방향

#### PgBouncer 모드
- `transaction` 모드 사용

#### 애플리케이션 풀 크기
- 애플리케이션 측 `pool_size`를 CPU 코어 수 기준으로 제한
- 제안 기준: `CPU 코어 수 * 2`

#### 유휴 트랜잭션 세션 정리
- PostgreSQL 설정:
  ```text
  idle_in_transaction_session_timeout=30s
  ```
- 장시간 열린 트랜잭션으로 인한 좀비 커넥션 방지 목적

### 운영상 시사점

- `max_connections`를 단순히 늘리는 방식은 근본 해결이 아닐 수 있음
- 커넥션 수요가 많은 환경에서는 PgBouncer 기반 멀티플렉싱이 더 안정적인 선택지로 평가됨

### 주의사항

- 이번 자료에는 워크로드 유형, DB 스펙, 실제 동시 접속 수, PgBouncer 상세 설정값은 포함되어 있지 않습니다.
- `transaction` 모드 사용 시 세션 상태 의존 로직이 있는 애플리케이션은 별도 검토가 필요할 수 있으나, 이에 대한 상세 내용은 원문에 없습니다.

