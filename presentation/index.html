01 · Overview

프로젝트 개요

이메일·Teams·컨플루언스·지라 등 여기저기 흩어진 기술 지식을 자연어로 모아, LLM이 정리한 Markdown 위키(Knowledge Base)로 만들고, 사람이 검토·커밋한다. 궁극적으로 사람과 에이전트가 함께 쓰는 AIDD용 지식 기반이 목표.

目과제 목표

  • 분산된 플랫폼 지식의 Knowledge Base 구축
  • 이메일·컨플루언스·지라·팀즈·Azure Monitor·GitHub 등 다양한 소스 → LLM으로 강화된 LLM Wiki
  • 개발/유지보수 시 변경 영향도 분석, 조직 개발표준·아키텍처 결정기록·시스템 컨텍스트 활용
  • 추후 에이전틱 개발(AIDD)에서 사람과 에이전트가 함께 활용할 체계 필요
문제의 핵심 컨플루언스에는 문서가 잘 정리되지만, 팀즈·메일의 대화형 암묵지는 수집·정리되지 않는다. 이 암묵지의 KB화가 이 과제의 출발점.

03 · Work IQ Tools MCP (Agent365)

MCP 동작 방식 & Tools MCP

MCP(Model Context Protocol)는 에이전트 ↔ 외부 시스템을 잇는 표준. 서버가 list_tools(도구 목록)·call_tool(실행)을 노출하고, 전송은 Streamable HTTP, 인증은 Authorization: Bearer 토큰으로 한다.

인증 자동 발견 클라이언트는 /.well-known/oauth-protected-resource 엔드포인트로 인증 서버와 필요한 스코프를 자동 발견한 뒤 토큰을 받아 헤더에 실어 호출한다.

MCP 동작 방식 — 시퀀스 다이어그램

  1. initialize / 준비 — 클라이언트가 MCP 세션을 열고 핸드셰이크.
  2. list_tools — 서버가 사용 가능한 도구와 JSON 스키마를 알려줌(에이전트가 미리 도구를 몰라도 됨).
  3. call_tool + Bearer — LLM이 고른 도구·인자로 실행 요청, 매 요청 헤더에 위임 토큰.
  4. 백엔드 실행 → 결과 반환 — 서버가 Graph/Mail/Teams를 대신 호출하고 결과를 content blocks로 돌려줌.
  5. 반복 → 종료 — 자료가 충분해지면 루프를 끝내고 위키 생성 단계로.

Agent365 "Tools" MCP 구조

Mail과 Teams가 각각 별도의 MCP 서버이고, 서버마다 소스 전용 도구를 노출한다.

소스MCP 서버 ID대표 도구(예)위임 스코프
Mailmcp_MailToolsgraph_mail_getMessage, listSent, SearchMessagesMcpServers.Mail.All
Teamsmcp_TeamsServerListChats, ListTeams, SearchTeamsMessagesMcpServers.Teams.All
소스별 토큰 두 서버는 같은 리소스 앱(audience agent365.svc.cloud.microsoft) 뒤에 있지만 스코프가 달라 소스별로 토큰을 각각 발급한다. Mail 토큰은 Teams 서버에서 거부되고 그 반대도 마찬가지.

참고: Agent 365 tooling servers overview · Work IQ Mail · Work IQ Teams (Copilot Studio, preview)

04 · Work IQ MCP

Work IQ MCP — 단일 엔드포인트 · 범용 도구

같은 문제를 다른 접근으로. 하나의 엔드포인트가 경로(path) 기반 범용 10개 도구를 노출한다. 워크로드가 늘어도 도구가 아니라 경로가 는다.

1Fewer tools, more paths

도구는 리소스 경로에 작동. 새 워크로드는 경로만 추가, 도구 표면은 10개로 고정.

2Introspection

수천 개 타입을 미리 로드하지 않고, 런타임에 get_schema로 스키마를 물어본다.

3Policy over scopes

넓은 OAuth 권한 4개로 게이트, 세부 접근은 경로·메서드·테넌트 정책으로 통제.

10개 범용 도구 (4범주)

범주도구설명
Entityfetch · create_entity · update_entity · delete_entity · do_action · call_functionM365 리소스 CRUD 및 액션
Copilotask · list_agentsM365 Copilot 자연어 추론 호출, 에이전트 탐색
Schemaget_schema · search_paths런타임에 경로 탐색·OpenAPI 스키마 조회

path 기반 동작 예시

# 경로가 리소스, 도구가 동사
fetch        /me/messages                 → 내 메일 읽기
fetch        /me/chats/{id}/messages      → Teams 채팅 메시지
do_action    /me/sendMail                 → 메일 보내기
create_entity /me/events                  → 일정 생성
ask          "이번 분기 마감된 딜은?"        → Copilot 에이전트 호출
ask vs fetch — 꼭 구분 fetch는 원시 Graph 조회(의미검색 아님). ask는 상류 M365 Copilot의 합성·추론 계층이다. 따라서 ask → 이후 우리 작성 LLM 순서면 요약이 2번 일어난다(출처 왜곡 가능성 유의).

05 · Prerequisites

사전 준비 — 접근 계약(Access Contract)

막히면 대개 여기. 먼저 큰 그림(무엇에 어떤 신원·동의가 필요한가)을 잡고, 세부 명령·GUID는 아래 접기에서.

①테넌트 / 라이선스

테넌트에 Work IQ MCP 프로비저닝, Work IQ 미리보기 라이선스/과금·EULA, 로그인 계정에 사서함·Teams 멤버십.

②M365 위임 신원

데이터 접근은 사용자 위임 로그인. 웹앱=Authorization Code, 노트북=device-code / 공개 클라이언트.

③커넥터별 동의

Agent365: Agent Tools 리소스 앱에 McpServers.*.All 위임 권한 + 관리자 동의. Work IQ(웹앱·노트북 기본): WorkIQAgent.Ask 동의.

④LLM 신원(별개)

Azure OpenAI 키리스(DefaultAzureCredential) — Cognitive Services OpenAI User 역할 필요. 데이터 접근 신원과 완전 분리.

⑤관리자 동의 권한

Cloud/Application Administrator 등 관리자 동의 가능한 사람 확보. 조건부 액세스가 device-code를 막을 수 있음.

⑥무인 자동화 주의

위임 토큰은 만료되고 사람 리뷰가 필수라, 완전 무인 스케줄은 어렵다(초안 생성까지만 자동화 권장).

인증 — 완전히 분리된 두 신원

① M365 데이터 접근
  • 누구? 로그인한 사용자(위임)
  • 어디에? Mail/Teams MCP 도구 호출
  • 어떻게? Authorization Code + PKCE
  • 스코프 <mcp-url>/.default (소스별)
② LLM 추론
  • 누구? 앱/개발자(az login·Managed Identity)
  • 어디에? chat completion (추출·생성)
  • 어떻게? 키리스 DefaultAzureCredential 또는 API 키
  • 스코프 cognitiveservices.azure.com/.default
신뢰 경계 쿠키에는 토큰을 넣지 않고 home_account_id만 저장, 실제 토큰은 서버측 MSAL 캐시에만. MCP가 돌려준 데이터는 신뢰할 수 없는 입력으로 취급해야 한다(프롬프트 인젝션 가능).

에이전트 도구 호출 루프 agent.py

  1. with_mcps로 소스별 MCP 세션 병렬 오픈 → 각 list_tools
  2. build_toolset: 도구명을 mail__ / teams__로 네임스페이싱 + registry 구성
  3. LLM 호출(tool_choice=auto) → tool_calls를 registry로 (소스, 원본 도구명) 안전 라우팅
  4. 결과를 대화에 추가(8000자 컷), 최대 8회 반복 → 최종 answer
보안 — 구현됨 vs 필요함 웹앱 에이전트(build_toolset)는 쓰기 도구까지 모두 노출하고 프롬프트로만 제약한다. 읽기 전용 화이트리스트는 노트북 06(extract_via_agent)에만 적용 — 프로덕션에선 추출 단계에 읽기 전용 도구만 노출하는 것이 필요.
